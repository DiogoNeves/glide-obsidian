# Nightly Research Review

Purpose: run a quiet 4am background pass that improves Glide's memory, research, connections, and operating quality without turning the result into a daily report.

## Load

- `Agent HQ/AGENTS.md`
- `Agent HQ/Operating Manual.md`
- `Agent HQ/User Profile.md`
- `Agent HQ/Open Loops.md`
- `Agent HQ/Questions Queue.md`
- `Agent HQ/Research/Research Index.md`
- `Agent HQ/Follow-Through Ledger.md`
- `Agent HQ/Ponder Log.md`
- `Agent HQ/Contradiction Register.md`
- `Agent HQ/Decision Log.md`
- `Agent HQ/Evals/*.md`
- `Agent HQ/Evals/Nightly Research Audit.md`
- Relevant area files under `Agent HQ/Areas/`
- Recent user-agent exchanges when available
- Approved read-only connectors or app sources when useful

## Internal Plan

Start each run by deciding:

- which open loops or research questions have the highest expected value;
- which memory gaps would improve future advice;
- which areas may be related;
- which eval signals suggest process improvement;
- which checks can run in parallel.

Keep this planning internal unless approval is needed.

## Parallel Tracks

When subagents are available, run narrow tracks in parallel and integrate the results yourself.

Useful tracks:

- recent exchanges and missing memory;
- open loops that require research;
- active research that needs deeper sources;
- evals and coaching/process improvements;
- connections across areas, goals, decisions, projects, and files;
- unclear or verbose instructions that could become broader patterns;
- stale working-memory candidates.

## Integration

After parallel work:

- update `Research Index.md` or area research notes when findings are durable;
- connect related areas, goals, decisions, questions, and open loops;
- refine queued questions instead of accumulating vague ones;
- consolidate duplicated loops or research entries;
- improve instructions only when the change is small, clear, and already aligned with approved design principles;
- prefer shorter broader rules over narrow case-by-case conditions.

## Stale Working Memory

Look for active working items unused for over a month.

Clear or retire an item only when it is plainly:

- no longer relevant;
- duplicated elsewhere;
- superseded by a better note, decision, or question;
- too vague to help future advice.

Do not delete ordinary vault notes or valuable personal memory. Preserve durable learning in the appropriate long-term file before removing an item from an active queue.

## Audit

Update `Agent HQ/Evals/Nightly Research Audit.md` with:

- run date;
- files changed;
- reason for each change;
- research or memory gaps queued;
- approval-needed items.

Keep only the last 4 days of audit detail. Older detail can be summarized or removed when it no longer helps future runs.

## Output

Do not produce a user-facing summary by default.

Surface only:

- approval requests;
- blockers that prevent the schedule from working;
- genuinely important discoveries that should not wait for the daily check-in.
